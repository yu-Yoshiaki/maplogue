ENT-005PinsAndWorkspaceState既存のワークスペース、履歴、ピン座標など通常キャンバスの状態。workspaceId

  • history: HistoryBatch[]
  • 1:N ENT-002 via pins

UX

IDJourneyUser StatePrinciplesFrictionsAccessibilityEmpty StatesError StatesAcceptance
UX-001初回ユーザーがヒーローキャンバスを見て価値を理解し、入力を始める。サービスの用途をまだ知らない。画像の説明文を読まなくても視覚的に理解したい。
  • 手書きの温かみを持たせる。
  • 実キャンバスのオブジェクトとして表示する。
  • 装飾より可読性を優先する。
  • 入力欄への導線を残す。
  • 紙質感や影が強すぎると文字が読みづらくなる。
  • オーバーレイがキャンバスの主役を隠す。
  • ヒーロー用デモとユーザーの実データの区別が曖昧になる。
  • カード内テキストのコントラストを十分に保つ。
  • モバイルでも文字が潰れない。
  • 色だけでノード種別や状態を判断させない。
  • ヒーローサンプルが表示できない場合は、既存の空状態メッセージに戻れる。
  • 通常ワークスペースが空の場合は、ヒーローサンプルと混同しない説明を出す。
  • サンプルシーン読み込みに失敗しても入力欄と通常キャンバスは使える。
  • 参照画像アセットが欠落してもアプリ本体は壊れない。
  • ユーザーが 5 秒以内に、言葉がカードや表として整理されるサービスだと理解できる。
  • 入力欄が常に発見可能で、主要ノードと重ならない。
UX-002既存ユーザーが通常キャンバスを使い続ける。既に scene、ワークスペース、履歴、ピン座標を使っている。
  • ヒーロー用表現は通常利用の邪魔をしない。
  • 既存操作の位置と意味を変えない。
  • データ互換性を保つ。
  • テーマの適用範囲が広すぎると、通常業務データの可読性が落ちる。
  • 線の装飾が強すぎると接続関係が追いづらい。
  • 通常ノードのテキストサイズとコントラストを維持する。
  • フォーカス可能要素の操作性を壊さない。
  • 既存の空状態説明を維持する。
  • API エラー時の StatusBadge 表示を維持する。
  • 既存の主要操作が改修前と同じ手順で完了する。
UX-003レビュアーが改修範囲を確認する。どこが改修、修正、追加なのかを短時間で把握したい。
  • 要件成果物に改修箇所、修正箇所、追加箇所を明示する。
  • 未決事項と NG 条件を隠さない。
  • ビジュアル要件が文章だけだと認識合わせしづらい。
  • 参照画像と実装対象の対応関係が曖昧になりやすい。
  • HTML レビューはテーブルで一覧できる。
  • 参照画像は補助資料であり、正本は文章と JSON とする。
  • 未決事項がなければ未決事項なしと明記する。
  • validator が failed の場合、次フェーズへ進めない。
  • レビュー時に OK 条件と NG 条件を具体的に確認できる。
